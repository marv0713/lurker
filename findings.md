# 发现与决策

## 需求
- 项目目标是建立一个大趋势投资雷达系统，用于发现港股、美股、A 股中正在形成或已经验证的中长期趋势。
- 用户最初关注“半年翻倍股”，但系统不能简单变成“翻倍就买”，而是用翻倍股反推产业逻辑和趋势扩散。
- 第一版定位为自用研究工具，短期不以收益证明系统，而是验证发现能力和复盘质量。
- 用户希望先捕捉异动，再分析验证行情质量，确认后进入长期跟踪，并持续判断强化、过热或证伪。
- 用户熟悉股票有限，因此系统需要具备自动扩池能力。
- 用户认可 AI 早介入，但不认可 AI 静默否决或黑箱吞掉已触发信号。

## 研究发现
- 固定核心龙头池会提高信噪比，但单独使用会漏掉早期扩散机会。
- 更合理的第一版结构是“种子池 -> 行业/概念扩池 -> 规则验证 -> AI 早期归因 -> 主候选/次级线索/观察池”。
- A 股、美股、港股不应使用完全相同的市场策略。
- A 股更适合承担主发现职责，因为产业链映射、板块扩散和主题活跃度更适合做趋势发现。
- 美股小公司资料不易获得，第一版更适合使用核心龙头、行业 ETF、主题 ETF 和跨市场映射公司。
- 港股老千股、低流动性和壳属性噪音较多，第一版必须强过滤，主要作为映射和补充市场。
- 行业/概念分类扩池比价格联动扩池更适合第一版，因为可解释性更好，也更利于复盘。
- AI 归因应采用“新闻先解释异动，公告/财报再做验证”的混合输入方式。

## 技术决策
| 决策 | 理由 |
|------|------|
| 第一版使用 Python 本地项目 | 适合数据抓取、指标计算、AI 调用和本地定时任务 |
| 第一版使用 SQLite | 本地自用足够，后续可迁移 Postgres |
| 使用 YAML 管理主题、市场和评分配置 | 配置可维护，后续产品化时不用把规则写死 |
| 使用 `akshare` 作为 A 股起步行情源 | 低成本，适合快速验证 |
| 使用 `yfinance` 作为美股/港股起步行情源 | 获取便利，适合 MVP |
| A 股第一刀只直接扫描 `seed_symbols`，`seed_indexes` 和 `seed_etfs` 先保留为来源边界 | 指数/ETF 成分股接口差异更大，应在真实行情链路验证后单独硬化 |
| A 股第二刀优先展开核心指数，不先展开 ETF | 沪深 300、中证 1000、科创 50、创业板指接口相对明确，ETF 持仓接口更容易出现字段和可得性差异 |
| A 股第三刀展开主题 ETF 重仓股 | 通过 akshare 公募基金持仓接口获取 ETF 最新季度前 N 大重仓股，提升细分主题覆盖率，并在 resolved universe 中保留 `sources.etfs` 来源 |
| 系统定位为中长期趋势研究工具，不是按日交易信号器 | Universe 应保持相对稳定，便于复盘趋势强化、扩散、过热和证伪 |
| seed pool 解析应手动或定期刷新，日常扫描复用缓存 | 避免每日行情扫描都依赖成分股接口，也保留“某次报告基于哪版研究宇宙”的可追溯性 |
| 行情快照先使用文件存储，并保留存储接口抽象 | 第一版避免数据库复杂度，同时让后续切 SQLite/Postgres 不影响应用层流程 |
| Gemini API Key 优先本地 `key` 文件 | 自用场景下便于本机运行；`key` 文件必须保持本地私有并被 git 忽略 |
| 每日闭环使用 `lurker daily-job` | 将刷新行情、生成日报和落盘报告合成一个本地定时任务入口，避免 cron 直接调用 demo 或手工串命令 |
| 每日闭环失败不应中断整批报告 | A 股行情源会偶发连接断开，失败应进入 `failures` 和日报风险提醒，后续再做重试/备用源硬化 |
| A 股行情稳定性优先于速度 | 用户不需要快，因此默认采用 `Tushare -> AkShare/Eastmoney -> BaoStock` 多源 fallback；无 Tushare token 时跳过 Tushare，BaoStock 可兜底 Eastmoney 断开 |
| AI 只处理规则触发后的候选 | 控制成本和噪音 |
| 使用结构化 AI 输出 | 方便映射为分数、风险标签和证据标签 |
| 每日输出两层结果 | 主候选便于快速阅读，次级线索避免黑箱错杀 |
